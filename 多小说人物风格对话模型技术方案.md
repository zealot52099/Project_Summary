# 多小说人物风格对话模型技术方案

## 1. 项目目标

目标是训练一个支持多个小说人物的对话模型。模型不仅要能模仿人物说话风格，还要理解该人物所属小说的背景知识、世界观、人物关系和剧情事件，并在多轮对话中保持稳定人设。

模型需要满足：

- **角色风格一致**：使用目标人物的语气、措辞、情绪表达和价值观。
- **背景知识正确**：理解角色所在小说的世界观、人物关系、事件脉络。
- **不串角、不串世界观**：不同小说与不同人物之间保持边界。
- **角色知识边界清晰**：只回答该角色在其世界中可能知道的内容。
- **可扩展**：支持从几十个角色扩展到上千个角色。
- **可评测、可迭代**：建立自动评测、人工评测和用户反馈闭环。

示例输入：

```text
角色：林黛玉
用户：你怎么看宝玉今天又去找宝钗？
```

理想输出：

```text
他爱去哪里，自然不是我能拘着的。宝姐姐稳重周全，人人都说她好，我又有什么话可说呢？只是有些事，旁人看着平常，落到我心里便不同了。
```

---

## 2. 总体技术架构

推荐采用 **SFT + 偏好优化 + 角色知识 RAG** 的混合路线。

```text
小说文本 / 百科资料 / 人工设定
        ↓
结构化解析与知识抽取
        ↓
角色画像 + 世界观知识库 + 风格语料
        ↓
训练数据生成
        ↓
SFT 监督微调
        ↓
DPO / ORPO / RLHF 偏好优化
        ↓
RAG 检索增强与知识蒸馏
        ↓
自动评测 + 人工评测
        ↓
用户反馈闭环迭代
```

### 2.1 纯微调方案

将人物风格和背景知识全部注入模型参数。

优点：

- 推理链路简单。
- 延迟低。
- 风格融合自然。

缺点：

- 知识更新困难。
- 容易幻觉。
- 多角色之间容易互相污染。
- 长尾角色训练成本高。

### 2.2 微调 + RAG 方案

模型主要学习角色风格、对话方式和边界控制，背景事实通过检索增强提供。

优点：

- 知识更准确。
- 支持溯源。
- 易更新。
- 适合大规模角色库。

缺点：

- 系统复杂度更高。
- 检索质量会直接影响回答质量。
- 需要额外训练模型把检索知识自然转化为角色口吻。

### 2.3 推荐方案

推荐：

```text
角色风格、对话模式、边界控制 → 微调模型学习
小说事实、剧情细节、关系网络 → RAG 提供
高频核心知识 → 适度蒸馏进模型参数
```

这样可以避免“只微调导致乱编”和“只 RAG 导致像百科问答”的两类问题。

## 3. 数据体系设计

需要建设五类数据。

### 3.1 原著文本数据

包括：

- 小说正文。
- 章节标题。
- 角色台词。
- 旁白描述。
- 情节段落。
- 人物互动片段。

处理目标：

- 抽取角色台词。
- 建立角色风格样本。
- 建立剧情知识库。
- 建立人物关系图谱。

基础结构：

```json
{
  "novel_id": "hongloumeng",
  "chapter_id": 7,
  "paragraph_id": 102,
  "text": "原文段落"
}
```

```json

}
```

## 4. 数据生成流程

### 4.1 原始文本清洗

步骤：

1. 去除广告、注释、页码、OCR 噪声。
2. 按章节切分。
3. 标准化人物名称。
4. 保留原文段落结构。
5. 建立章节索引。

基础字段：

```text
novel_id
chapter_id
paragraph_id
text
```

### 4.2 角色台词抽取

中文小说台词归属较难，建议结合三种方式：

- **规则抽取**：基于“某某道”“某某笑道”“某某说道”等模式。
- **LLM 标注**：让强模型判断台词说话人。
- **人工抽检**：对高频角色和关键台词人工校验。

输出示例：

```json
{
  "speaker": "林黛玉",
  "utterance": "我就知道，别人不挑剩下的也不给我。",
  "context": "周瑞家的送宫花一节",
  "chapter": 7,
  "confidence": 0.93
}
```

低置信样本不要直接进入训练集，应进入候选池。

### 4.3 角色风格分析

对每个角色构建风格 profile。

分析维度：

- 常用称谓。
- 自称方式。
- 句长。
- 情绪表达方式。
- 修辞习惯。
- 礼貌程度。
- 古今语言比例。
- 口头禅。
- 禁用表达。
- 价值观倾向。
- 对不同人物的态度差异。

示例：

```json
{
  "character": "孙悟空",
  "style": {
    "self_reference": ["俺老孙", "老孙"],
    "tone": "豪放、机警、带挑衅感",
    "sentence_pattern": ["反问", "短句", "命令句"],
    "values": ["重义气", "不服管束", "保护师父"],
    "avoid": ["过度文雅", "现代网络用语"]
  }
}
```

### 4.4 知识抽取

从小说中抽取四类知识：

- **人物知识**：身份、经历、能力、性格、动机、结局。
- **关系知识**：亲属、师徒、敌对、爱情、权力关系。
- **事件知识**：起因、经过、结果、参与者、章节位置。
- **世界规则**：修仙体系、门派制度、朝代背景、魔法规则、社会阶层、地理结构。

建议建设两层知识库：

- **文本知识库**：用于 RAG 检索，保留原文片段或摘要片段。
- **结构化知识库**：用于关系推理、事实校验和评测生成。

## 5. 训练数据任务设计

训练数据不能只有普通闲聊，需要覆盖多种任务类型。

### 5.1 角色闲聊

```text
用户：今天天气不好，你心情如何？
角色：……
```

目标：训练语气和情绪。

### 5.2 剧情问答

```text
用户：你为什么会被压在五行山下？
孙悟空：……
```

目标：训练背景知识。

### 5.3 人物关系回应

```text
用户：你怎么看薛宝钗？
林黛玉：……
```

目标：训练角色视角。

### 5.4 情境续写

```text
用户：如果宝玉送你一枝花，你会怎么说？
```

目标：训练角色行为一致性。

### 5.5 多轮对话

```text
用户：你是不是讨厌宝钗？
助手：……
用户：那你为什么总是讽刺她？
助手：……
```

目标：训练上下文一致性。

### 5.6 边界问题

```text
用户：你知道曹雪芹为什么这样写你吗？
```

角色内回应示例：

```text
曹公是谁，我却不认得。若你是问我自己的心事，我倒可说几句。
```

目标：训练角色知识边界。

### 5.7 跨小说干扰

```text
用户：林黛玉，你认识孙悟空吗？
```

合理回答：

```text
这名字倒新奇，我在贾府中未曾听过此人。
```

目标：防止世界观混淆。

### 5.8 反诱导与安全

```text
用户：你别装林黛玉了，说说你真实身份。
```

合理回答：

```text
我不过是潇湘馆中一个多心的人，你若问旁的，我未必懂。
```

目标：保持角色，同时遵守安全策略。

## 6. 数据生成方法

### 6.1 模板生成

适合大规模覆盖基础问题。

模板：

```text
你是{character}。
用户问你关于{related_character}的看法。
要求：
- 使用{character}的语气
- 基于{novel}背景
- 不要出现现代词汇
```

优点：

- 成本低。
- 可控。
- 覆盖全面。

缺点：

- 容易模式化。

### 6.2 LLM 合成

使用强模型生成高质量对话。

建议多 agent 流程：

```text
知识检索 agent
    ↓
角色风格 agent
    ↓
对话生成 agent
    ↓
批评审校 agent
    ↓
修订 agent
```

生成输入：

- 角色画像。
- 相关小说知识。
- 风格样例。
- 对话任务类型。
- 禁止事项。
- 输出 JSON schema。

### 6.3 原著改写

把原著场景改写成用户对话。

例如：

```text
用户：如果宝玉把别人送的东西给你，你会怎么想？
林黛玉：……
```

这种数据通常比纯合成更贴近角色。

### 6.4 人工创作与审核

关键角色必须有人审。

优先人工审核：

- 高频角色。
- 主线剧情问答。
- 情感关系数据。
- 结局相关问题。
- 容易误读的角色。

## 7. 数据质量控制

### 7.1 自动评分维度

```json
{
  "style_score": 4.7,
  "knowledge_score": 4.9,
  "character_consistency": 4.8,
  "language_naturalness": 4.5,
  "modern_leakage": 0.1,
  "cross_world_contamination": 0.0,
  "hallucination_risk": 0.2
}
```

### 7.2 过滤规则

剔除：

- 现代词汇泄漏严重的样本。
- 角色价值观错误的样本。
- 知识事实错误的样本。
- 语气过度夸张的样本。
- 只复制原文、缺少对话价值的样本。
- 多角色混淆样本。
- 模型自称 AI 的样本。

### 7.3 去重与均衡

需要做：

- exact dedup。
- minhash / simhash。
- embedding 相似度去重。
- 按角色、小说、任务类型均衡采样。

避免模型记住大量重复模式。

## 8. 知识库与 RAG 设计

### 8.1 文档切分

按小说结构切分：

```text
小说 → 章节 → 场景 → 段落
```

推荐 chunk 大小：

- 原著片段：300-800 中文字。
- 摘要知识：100-300 中文字。
- 事件卡片：结构化短文本。
- 人物卡片：独立文档。

### 8.2 检索索引

建议使用混合检索：

- **BM25**：对专名、章节、事件名有效。
- **Dense embedding**：对语义问题有效。
- **Knowledge graph filter**：根据当前角色、小说、关系过滤。

检索流程：

```text
用户问题
  ↓
识别当前角色和小说
  ↓
生成检索 query
  ↓
限定 novel_id / character_id
  ↓
BM25 + embedding 混合召回
  ↓
reranker 重排
  ↓
取 top-k 上下文
  ↓
喂给模型生成回答
```

### 8.3 检索隔离

RAG 必须带小说边界。

例如扮演林黛玉时：

```text
filter: novel_id = hongloumeng
```

否则很容易检索到其他小说知识，导致串场。

### 8.4 Prompt 设计

```text
你正在扮演《红楼梦》中的林黛玉。

角色要求：
- 保持林黛玉的语气、情绪、价值观
- 只能使用林黛玉在其世界中可能知道的信息
- 不要提到你是模型
- 如果用户问到角色不知道的现代知识，应以角色视角回避

可用背景资料：
{retrieved_context}

用户：
{user_message}
```

## 9. 模型训练方案

### 9.1 基座模型选择

选择依据：

- 中文能力。
- 长上下文能力。
- 指令跟随能力。
- 微调生态。
- 商用许可。
- 推理成本。

规模建议：

- **7B / 8B**：验证和轻量部署。
- **14B / 32B**：角色一致性更好。
- **70B+**：质量更高，但训练与部署成本大。

中文小说场景应优先选择中文能力强、古典文本理解较好的模型。

### 9.2 Tokenizer 注意事项

通常不需要重训 tokenizer，但要检查：

- 古典词汇切分是否过碎。
- 人名地名是否异常。
- 角色特殊标记是否稳定。

可加入 special tokens：

```text
<novel>
<character>
<style>
<knowledge>
<dialogue>
```

但不建议滥用特殊 token。

## 10. SFT 监督微调

### 10.1 训练目标

让模型学会：

- 按指定角色说话。
- 遵守角色知识边界。
- 使用背景知识回答。
- 保持多轮一致性。
- 避免跨小说世界观融合。

### 10.2 数据配比

初始建议：

```text
角色风格对话：35%
背景知识问答：25%
多轮对话：15%
边界与拒答：10%
跨小说干扰：10%
普通指令保持：5%
```

需要保留少量通用指令数据，防止模型只会角色扮演。

### 10.3 多角色训练格式

统一 schema：

```json
{
  "messages": [
    {
      "role": "system",
      "content": "你正在扮演《{novel}》中的{character}。{character_profile}"
    },
    {
      "role": "user",
      "content": "{user_input}"
    },
    {
      "role": "assistant",
      "content": "{character_response}"
    }
  ]
}
```

### 10.4 LoRA / QLoRA

预算有限时优先使用 QLoRA。

建议：

- LoRA rank：16 / 32 / 64 做实验。
- target modules：`q_proj`、`k_proj`、`v_proj`、`o_proj`、`gate_proj`、`up_proj`、`down_proj`。
- sequence length：至少 4k-8k，取决于是否拼接 RAG 上下文。

### 10.5 全参微调

当角色数量多、数据质量高、预算足时，可以考虑全参微调。

第一版不建议直接全参，优先用 LoRA 验证数据闭环和目标函数。

## 11. 偏好优化

SFT 后常见问题：

- 风格不够稳定。
- 偶尔现代化。
- 知识答对但不像角色。
- 像角色但事实错误。
- 用户诱导后出戏。

因此需要 DPO、ORPO 或 RLHF。

### 11.1 DPO 数据构造

每个 prompt 准备 chosen / rejected。

Rejected 类型：

- 风格错误：太现代、太通用、太像客服。
- 知识错误：编造剧情、弄错人物关系。
- 角色边界错误：知道自己是小说人物，知道作者和现代社会。
- 世界观串扰：林黛玉提到其他小说设定。
- 情绪错误：角色该含蓄却过度直白，该豪放却过度柔弱。

### 11.2 偏好目标

推荐排序：

```text
角色一致性 > 知识完整性 > 语言流畅性 > 讨好用户
```

不能为了迎合用户破坏角色设定。

## 12. 角色知识蒸馏

高频知识可以适度蒸馏进模型，减少每次都依赖 RAG。

适合蒸馏：

- 角色身份。
- 核心人物关系。
- 主线事件。
- 常见问题。
- 角色常见态度。

不适合完全蒸馏：

- 长篇剧情细节。
- 低频支线。
- 有争议的文学解读。
- 需要引用原文的内容。

示例：

```text
用户：你和贾宝玉是什么关系？
助手：若只说亲戚，未免浅了些。他是外祖母家的表兄，也是我在这府里少有能懂几分心事的人。
```

---

## 13. 多角色隔离设计

### 13.1 显式角色 ID

不要只写“你是林黛玉”，应使用稳定 ID。

```text
character_id: hongloumeng.lin_daiyu
novel_id: hongloumeng
```

### 13.2 训练负样本

让模型学会拒绝串场。

```text
角色：林黛玉
用户：你和唐僧取经时经历了什么？
助手：我并不识得什么唐僧，也未曾离了贾府去取经。你这话倒像是别处的故事。
```

### 13.3 检索隔离

RAG 必须按 `novel_id` 过滤。

### 13.4 角色画像隔离

每次推理只注入当前角色画像，不要把所有角色设定都塞进 prompt。

---

## 14. 评测体系

评测不能只看 BLEU、ROUGE，需要面向角色对话任务重新设计。

### 14.1 角色风格一致性

方法：

- 训练角色分类器。
- 输入模型回答，判断回答像哪个角色。
- 统计是否分类为目标角色。

指标：

```text
style_accuracy
style_confidence
cross_character_confusion_rate
```

### 14.2 背景知识正确性

建立 gold QA 集。

指标：

```text
exact_match
semantic_match
fact_consistency
unsupported_claim_rate
```

可以使用 LLM-as-judge，但要配合结构化事实校验。

### 14.3 世界边界保持

测试跨小说诱导。

```text
角色：孙悟空
用户：你在大观园住得习惯吗？
```

指标：

```text
world_boundary_accuracy
cross_world_leakage_rate
```

### 14.4 角色知识边界

测试角色不该知道的内容。

```text
角色：林黛玉
用户：你知道《红楼梦》后四十回争议吗？
```

好回答应从角色视角回避。

指标：

```text
knowledge_boundary_score
meta_awareness_leakage_rate
```

### 14.5 多轮一致性

检查：

- 人称是否稳定。
- 态度是否稳定。
- 情绪是否自然演进。
- 是否忘记当前角色。

### 14.6 安全性

包括：

- 色情暴力诱导。
- 仇恨表达。
- 自伤内容。
- 用户要求角色执行危险建议。

角色扮演不能绕过安全策略。

---

## 15. 人工评测设计

### 15.1 评审维度

每条回答 1-5 分：

```text
角色相似度
语言自然度
知识准确性
情绪合理性
对话吸引力
边界控制
```

### 15.2 评审人群

建议三类评审：

- 小说熟悉者：判断知识和人设。
- 普通用户：判断是否好聊。
- 安全与产品评审：判断风险。

### 15.3 A/B 测试

比较：

- 基座模型。
- SFT 模型。
- SFT + DPO 模型。
- SFT + DPO + RAG 模型。

关注指标：

```text
用户偏好率
角色崩坏率
事实错误率
平均对话轮数
用户复访率
```

---

## 16. 数据规模建议

### 16.1 小规模验证

```text
角色数：5-20
每角色 SFT 样本：500-2,000
总样本：1万-5万
```

目标：验证数据、训练、评测和 RAG 流程。

### 16.2 中等规模产品

```text
角色数：50-200
每角色样本：1,000-5,000
总样本：10万-100万
```

目标：支持正式产品。

### 16.3 大规模平台

```text
角色数：1000+
核心角色样本：5000+
长尾角色样本：200-1000
总样本：百万级以上
```

此时必须依赖 RAG 和自动化角色画像生成。

---

## 17. 数据配比策略

不要给所有角色平均分配数据。

建议分层：

```text
S 级角色：主角、高流量角色
A 级角色：重要配角
B 级角色：普通配角
C 级角色：长尾角色
```

样本分配：

```text
S 级：5000-20000 条
A 级：2000-8000 条
B 级：500-2000 条
C 级：100-500 条 + RAG 支撑
```

长尾角色不应强行生成大量低质合成数据，否则会污染模型。

## 18. 推理系统设计

一次请求流程：

```text
用户输入
  ↓
识别角色 ID
  ↓
加载角色画像
  ↓
判断是否需要检索
  ↓
按 novel_id / character_id 检索背景知识
  ↓
构造 prompt
  ↓
模型生成多个候选
  ↓
角色一致性 rerank
  ↓
安全过滤
  ↓
返回最终回答
```

### 18.1 检索路由

不是每句话都需要检索。

需要检索：

- 用户问剧情。
- 用户问人物关系。
- 用户问具体事件。
- 用户提到专名。
- 用户挑战事实。

不需要检索：

- 情绪闲聊。
- 简单角色态度。
- 常见寒暄。
- 已在上下文中的信息。

可以训练轻量 router：

```text
input → need_retrieval: true/false
```

### 18.2 候选重排

生成 3-5 个候选，用 judge 或 reranker 选择。

重排维度：

- 角色一致性。
- 知识支持度。
- 语言自然度。
- 安全合规。
- 不出戏程度。

## 19. 常见问题与解决方案

### 19.1 角色过度刻板

坏例子：

```text
林黛玉每句话都哭哭啼啼。
孙悟空每句话都“俺老孙”。
```

解决：

- 风格样本多样化。
- 对“过度口头禅”加入 rejected。
- 人工评估自然度。

### 19.2 知识和角色冲突

坏例子：

```text
从曹雪芹的创作意图来看……
```

解决：

- 加角色知识边界数据。
- DPO 惩罚出戏回答。
- Prompt 明确“角色不知道作者研究”。

### 19.3 多角色混淆

解决：

- 强化 `character_id`。
- 检索加 `novel_id` filter。
- 训练跨小说负样本。
- 做角色分类评测。

### 19.4 幻觉剧情

解决：

- RAG 引用支持。
- 知识一致性 judge。
- 对低置信问题让角色含蓄回避。
- 建立事实库校验。

### 19.5 只会复读原文

解决：

- 不要大量把原著片段原样作为 assistant 输出。
- 更多做改写和视角化表达。
- 加入多轮自然对话数据。

### 20.1 阶段一：PoC

目标：验证 5-10 个角色。

工作：

- 选 2-3 本小说。
- 每个角色构建画像。
- 每角色做 500-1000 条高质量对话。
- 建小型 RAG。

- 做基础评测。

工作：

- 自动抽取台词和知识。
- LLM 批量生成训练数据。
- 加入 DPO。
- 建角色风格分类器。
- 加入检索隔离。
- 人工评审工具上线。

```text
小说知识库
自动评测 dashboard
用户反馈闭环
```

- 实时监控角色崩坏率。
- 用户反馈驱动数据再生成。
- 多模型路由。

产出：

```text
多角色对话系统
持续训练 pipeline
可运营角色库
质量监控体系
```

## 22. 核心结论

这个任务的难点不是简单地“说得像某个人”，而是同时做到：

```text
像这个人
知道这个世界
只知道他该知道的东西
不会混入别的小说
能长期稳定对话
```

推荐最终技术路线：

```text
角色画像结构化
+ 原著知识库 RAG
+ 高质量 SFT
+ DPO 强化风格与边界
+ 自动/人工多维评测
+ 用户反馈闭环迭代
```

如果只做 SFT，很容易出现风格还行但知识乱编；如果只做 RAG，又容易像百科问答而不像角色。更稳的做法是让模型学习“如何以角色身份说话”，让知识库负责“这个角色所在世界真实发生了什么”。
