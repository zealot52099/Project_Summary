# VIVO GUI Agent 项目总结

## 1. 项目背景与目标

VIVO GUI Agent 项目面向手车互联场景，目标是在车机语音入口中，通过小 P 语音理解用户指令，并操控 VIVO 手机镜像完成微信、飞书、通知、导航等任务。项目核心不是单纯的语音指令分发，而是将语音、车机端、手机镜像、截图、云端多模态模型、GUI Agent、业务执行链路串成一个可运行、可追踪、可评测的端到端系统。

项目 V1 重点覆盖单轮场景，暂不考虑复杂小 P+ 多轮体验。主要需求包括：

- 小 P 操控 VIVO 手机发送微信消息。
- 小 P 控制手机通知消息的筛选、播报、打开、屏蔽与恢复。
- 小 P OCR 识别 VIVO 手机镜像中的地址信息，并支持语音发起导航。
- 基础能力：打开/关闭手机应用，接听/拒接音视频电话。
- 扩展方向：发飞书消息、进入飞书消息页面、读语音消息、联系人澄清、任务状态继承。

从算法侧看，项目核心目标是构建一个可在真实手机页面中执行任务的 GUI Agent：输入用户 query、页面截图、历史动作与上下文，输出可执行 action，并通过车机与手机镜像链路完成点击、长按、输入、滑动、返回、打开应用、等待、追问用户、完成任务等操作。

## 2. 总体架构

整体方案可以概括为：语音入口 + 意图分发 + 截图上传 + 多模态理解 + GUI Agent 决策 + 端侧执行 + 状态检测与回流。

### 2.1 端到端链路

1. 用户通过小 P 语音发起请求，例如“给张三发微信晚上去吃烧烤”。
2. 语音车机端接收 ASR/NLU 结果，并调用云端 NLU 大模型识别意图。
3. NLU 输出 APAF 或对应 action function，由语音平台分发到车机端或执行引擎。
4. 车机端通过 CarlinkMirror / mirror app 获取手机镜像状态，必要时调用截图能力。
5. 截图经客户端预处理后上传 OSS，生成可访问 URL。
6. 云端算法服务读取截图 URL，调用 GUI Agent 或 OCR / 页面状态检测模型。
7. Agent 输出下一步动作，如 click、long_press、input、wait、call_user、finish 等。
8. 平台将 action 下发到车机端，车机端通过手机镜像反控能力执行动作。
9. 每轮执行后重新截图，进入下一轮观察、规划、动作生成与执行。
10. 任务完成后输出 finish / summary / TTS 话术，或者进入异常处理、追问澄清、用户确认等状态。

### 2.2 主要参与模块

- 语音平台：接收车机端识别结果，调用 NLU，处理 APAF 与模型输出，并向车机端分发动作。
- 语音车机端：接收感知信息，和 CarlinkMirror / mirror app 交互，调用截图服务，执行平台下发动作。
- 应用车端 / CarlinkMirror：提供手机镜像、截图、反控、应用打开/关闭、连接状态等能力。
- System UI / AI Bar：提供通知消息展示、通知处理、屏蔽/恢复等能力。
- 感知服务：提供 VIVO 镜像连接状态、微信音视频电话状态等感知信息。
- 算法服务：负责 NLU 分发、OCR 地址识别、页面状态识别、GUI Agent 动作决策、后训练和评测。
- 导航能力：接收 OCR 地址结果或 POI 信息，调用导航搜索并发起导航。

## 3. 技术路线：检测 + 场景化能力增强

微信发消息方案采用“检测 + 场景化能力增强”的架构模式，而不是完全依赖一个通用 GUI Agent 端到端硬跑所有情况。

核心思想：

- 使用页面状态检测模型判断当前局部场景，例如是否在微信首页、搜索页、对话框页面、是否有弹窗、是否处于键盘状态等。
- 使用通用 GUI Agent 模型预测下一步动作，直到 finish 或 error。
- 对局部关键场景做规则、SFT 或后处理增强，例如粘贴输入、联系人澄清、输入框状态确认、发送前二次确认。
- 通过 history 分析判断模型输出是否需要纠正，必要时进行 action correction 或 replan。
- 任务回到主流程后继续由 Agent 决策下一步动作。

该方案的主要收益是：在保持通用 GUI Agent 能力的同时，把微信/飞书等强产品约束场景中的关键分支做稳定增强，降低纯通用模型在真实 APP 页面中出错的概率。

## 4. 微信发消息 GUI Agent 方案

### 4.1 主要挑战

微信发消息任务的难点集中在以下几类：

- 起始页面不可控：用户可能停留在手机桌面、微信首页、微信非消息页、搜索页、对话框页、其他 APP、分屏页或弹窗页。
- 页面状态需要准确识别：必须判断当前是否在微信、微信哪个页面、是否存在弹窗遮挡、是否需要回到微信首页。
- 联系人搜索与澄清复杂：搜索结果可能有联系人、群聊、小程序、网络结果，且可能出现多个同名联系人或只有群聊。
- 微信输入方式受限：不能直接 type 输入文字，需要通过粘贴板 + 长按输入框 + 点击粘贴的方式完成输入。
- 发送前需要确认：进入目标对话框并输入内容后，需要识别当前对话对象和输入内容，再向用户确认是否发送。
- 定制化需求较多：大量产品体验要求不完全符合通用 GUI Agent 的自然能力，需要通过检测、规则、后处理和定制数据增强实现。

### 4.2 执行流程

典型任务：“给张三发微信晚上去吃烧烤”。

1. 接收用户 query，NLU 分发为 SendMessage / 发微信任务。
2. 获取当前手机镜像截图并上传 OSS。
3. 页面状态检测：判断是否在微信、是否有弹窗、是否处于首页/搜索页/对话框页等。
4. 若不在微信或状态不合适，执行 open_app / app_home / go 等动作回到微信消息页。
5. 在微信首页点击搜索框。
6. 将联系人名称转为拼音或指定搜索词，写入粘贴板。
7. 长按搜索框，点击“粘贴”，等待搜索结果。
8. 分析搜索结果：
   - 单一联系人：默认点击。
   - 多个联系人：call_user 追问用户选择第几个。
   - 无联系人但有群聊：按群聊逻辑处理。
   - 多个群聊：call_user 追问。
   - 无结果：task_failed / call_user。
9. 进入对话框后识别对话对象名称，确认是否与目标一致。
10. 将消息内容写入粘贴板。
11. 长按输入框，点击“粘贴”。
12. 检测输入框是否已有内容，向用户确认“是否发送”。
13. 用户确认后，点击发送按钮。
14. 发送完成后 finish，并播报“已发送”。

### 4.3 关键模块

#### 页面状态检测

用于识别当前页面类别和局部状态，决定 Agent 是否继续执行、回首页、等待、追问或异常处理。

微信页面分类包括：

- 微信与其他 APP 分屏页面
- 微信首页
- 微信首页非消息页
- 微信搜索页面
- 对话框页面
- 微信其他页面
- 桌面
- 飞书页面
- 其他页面
- 弹框页面

后续在飞书场景中也定义了 7 类页面：飞书首页、飞书首页非消息页、飞书搜索页面、飞书对话框页面、飞书其他页面、飞书与其他 APP 分屏页面、其他页面。

#### 追问澄清模块

触发条件包括：

- 搜索结果出现多个联系人。
- 搜索结果出现多个群聊。
- 用户任务没有指定联系人，例如“发消息xxx”但当前不在联系人对话框。
- 搜索结果不确定，需要用户选择第几个。
- 当前页面对话对象与目标不一致。

典型话术包括：

- “找到多个联系人，你要发给第几个？”
- “当前不在目标联系人对话框，请问你想发给谁？”
- “确认发送吗？”

#### Action 纠正与 Replan

当执行历史与当前页面状态不一致时，需要纠正动作或重新规划。

触发场景包括：

- 页面未找到目标。
- 多次执行后页面不变。
- 误跳转到其他 APP。
- 弹窗遮挡导致原计划无法执行。
- 原计划与当前 observation 严重不符。

较稳健的做法是维护“当前生效计划 Active Plan”，当出现严重偏离时显式废弃旧计划，并生成新计划。旧 history 不删除，但在上下文中标记为 obsolete，后续动作以最新 active plan 为准。

#### 执行后检测

每次 action 后通过截图变化、页面状态检测、输入框状态检测等判断动作是否成功。例如：

- 点击联系人后是否进入对话框。
- 长按输入框后是否出现“粘贴/全选”悬浮框。
- 点击粘贴后输入框是否有内容。
- 点击发送后消息是否出现在聊天记录中。

## 5. 动作空间与接口规范

### 5.1 手机 GUI Agent 动作

VIVO 微信 GUI Agent 使用的动作包括：

- `click(x, y)`：坐标点击。
- `long_press(x, y)`：长按。
- `scrollTo(x, y)`：容器绝对滑动。
- `scrollBy(x, y)`：容器相对滑动。
- `drag(fromX, fromY, toX, toY)`：拖拽。
- `input(text)`：输入文本，微信场景中主要用于写入粘贴板。
- `go(direction)`：页面前进/后退，一般用于返回上一页。
- `open_app(app)`：拉起应用。
- `close_app(app)`：关闭/杀死应用。
- `app_home()`：回到微信或飞书首页。
- `wait(duration)`：等待页面变化或弹窗消失。
- `call_user(text)`：向用户追问或请求确认。
- `summary(text)`：返回总结结果。
- `finish(text)`：任务完成。
- `task_impossible()` / error：任务无法完成。
- `name(content)`：返回当前对话对象名称。

微信场景的特殊点是：不能直接向输入框 type 文字，而是通过“写入粘贴板 + 长按输入框 + 点击粘贴”完成输入。

### 5.2 AI 浏览器外部依赖动作

AI 浏览器侧也定义了类似动作空间，包括：

- 获取 DOM：`getDomTree()`
- 元素点击：`click(jspath)`
- 坐标点击：`click(x, y)`
- 滑动：`scrollTo(x, y)` / `scrollBy(x, y)`
- 拖拽：`drag(fromX, fromY, toX, toY)`
- 输入：`input(jspath, text, enter)` / `input(x, y, text, enter)`
- 前进后退：`go(direction)`
- 页面跳转：`jumpTo(url)`
- 主页：`home()`
- 刷新：`refresh()`
- 收藏：`favorite(action)`
- 非 action 透传：`summary`、`login`、`call_user`、`finish`

AI 浏览器规范中的记忆入参包括截图、HTML、turn_id、scene_id、action_history、reflection_history、planner_history、知识库 plan 等；输出包括 action_res、planner_res、action_plan 和更新后的历史。

## 6. 飞书 GUI Agent 方案

### 6.1 项目目标

飞书方向目标是实现手车互联发飞书 Agent，使用一个自研模型端到端完成发飞书消息、打开飞书对话框、飞书打电话、读语音消息等任务。

目标指标：

- 单步准确率：98%。
- 整体任务完成准确率：90%，若投入更多人力可尝试提升到 95%-97%。
- 单步推理耗时：
  - 图片压缩 + 传输低于 300ms。
  - Agent 推理低于 600ms。
  - 等待约 2s。
  - 总体单步推理时长不超过 3s。

### 6.2 发飞书消息流程

任务：“给 xx 发消息 xxx”。

步骤：

1. 进入飞书消息页面。
2. 搜索联系人 xx。
3. 若多个联系人或群组，触发 recall 追问。
4. 选择第 x 个。
5. 输入消息 xxx。
6. 确认发送。

如果用户只说“发消息 xxx”，且当前页面不是飞书联系人对话框，则需要追问联系人名称。

### 6.3 飞书页面分类

飞书页面状态定义为 7 类：

- 飞书首页
- 飞书首页非消息页
- 飞书搜索页面
- 飞书对话框页面
- 飞书其他页面
- 飞书与其他 APP 分屏页面
- 其他页面

数据多样性要求包括：

- 不同飞书账号与组织：XP 飞书、个人账号飞书、其他公司飞书。
- 黑色背景与白色背景各占一半。
- 键盘样式尽可能丰富。
- 覆盖飞书定制首页、搜索页、对话页、其他页和分屏页。

### 6.4 进入飞书消息页面数据设计

进入飞书消息页面任务按当前页面类别构造 action：

- 其他页面 / 分屏页面 / 飞书其他页面 / 飞书搜索页面 / 飞书对话框页面：执行 kill && open 飞书。
- 飞书首页非消息页：先点击消息按钮，再 finish。
- 飞书消息页面：直接 finish。
- 发消息但当前不在对话框页面：recall。
- 当前为对话框页面：执行输入消息流程。

训练/测试规模设计示例：

- XP 飞书首页：训练 40，测试 20。
- XP 飞书首页非消息页：训练 40，测试 20。
- 个人账号飞书首页：训练 10，测试 5。
- 个人账号飞书首页非消息页：训练 10，测试 5。
- 其他公司飞书首页：训练 20，测试 10。
- 其他公司飞书首页非消息页：训练 20，测试 10。
- 飞书搜索页面：训练 40，测试 20。
- 飞书对话框页面：训练 40，测试 20。
- 飞书其他页面：训练 40，测试 20。
- 飞书与其他 APP 分屏页面：训练 40，测试 20。
- 其他页面：训练 250，测试 125。

### 6.5 飞书风险点

- 各公司飞书首页 UI 定制化严重，需要通过数据分布扩增、网络爬取和线上迭代解决。
- 日程提醒等定制弹窗会干扰搜索和点击，需要数据生成与线上迭代。
- 键盘定制化严重，需要针对键盘做数据生成。
- 黑白主题、字体大小、飞书定制入口、联系人/群组搜索规则都需要覆盖。

### 7.2 弹窗遮挡识别

弹窗遮挡识别主要判断当前页面顶部是否出现 APP 消息横幅或其他遮挡。它对搜索、点击、输入等动作非常关键，因为弹窗可能遮挡搜索框、联系人或粘贴按钮。

结果演进：

任务是判断微信聊天对话框页面是否存在右侧带红点的未读语音消息。

结果：

- 0825：整体 54.55%，主要问题是不含未读语音样本误判。
- 0902 之后：含未读和不含未读均达到 100%。

### 7.4 输入框与键盘状态识别

关键子任务包括：

- 长按输入框后悬浮框是否包含“全选”。
- 当前是否处于语音输入模式，即底部是否出现“按住 说话”。
- 输入框内部是否有用户输入内容，且不能把粘贴浮窗、聊天气泡、按钮文字误判为输入内容。

结果：

- 键盘状态识别从 50% 提升到 100%。
- 输入框“全选”识别多轮达到 100%，1215 为 95.12%。
- 对话框输入框是否有字在 1208 达到 98.41%。

### 7.5 搜索结果 list 识别

搜索结果识别用于联系人/群聊澄清。输出结构要求按以下类别分类：

- 最常使用
- 联系人
- 群聊
- 小程序
- 搜索网络结果

```json
  "搜索网络结果": []
}
```

## 8. 模型训练与数据方案

### 8.1 模型选型

飞书技术方案中提到的模型候选包括：

- uitars1.5 7B。
- 后续可尝试在自研理解模型上进行轨迹训练。
- 微信方案中还涉及 3B 小模型做页面状态检测，7B 通用模型预测下一步动作。

### 8.2 输入输出设计

GUI Agent 输入：

- 用户指令 user_instruction。
- 当前截图。
- action_history。
- 可选 planner_history、reflection_history。
- 页面状态检测结果。
- 知识库 plan 或历史经验。

输出：

- Thought：中文简短描述下一步行动。
- Action：结构化动作，例如 click、type、app_home、wait、call_user、finish、name、long_press。

飞书 action_type 包括：

- click
- type
- finish
- app_home
- call_user
- wait
- task_impossible
- name
- long_press

action_des 可包括：

- 进入飞书首页
- 点击消息按钮
- 点击搜索按钮
- 输入 xxx
- 点击联系人 xx
- 点击发送
- 等待 x 秒
- 完成
- 发现 x 联系人 / x 群组
- 未发现联系人
- 当前不在飞书对话框页面
- 比对当前对话对象名称

## 9. 性能与耗时分析

### 9.1 未加速前微信场景耗时

场景一：“给张三发微信晚上去吃烧烤”：

- 视频总时长约 33s。
- Agent 算法合计耗时 7.6s。
- Agent 算法耗时占比：7.6 / 33 = 23%。
- 若不算 8 次 sleep，算法耗时占比约 30.4%。
- 平均每次 Agent 请求耗时：7.6 / 5 = 1.52s。
- 平均单 action 耗时：7.6 / 8 = 0.95s。

场景二：“给龙哥发微信说今天晚上吃烧烤”：

- 视频总时长约 42s。
- Agent 算法合计耗时 10.85s。
- Agent 算法耗时占比：25.83%。
- 若不算 13s sleep，算法耗时占比约 37.4%。
- 平均每次 Agent 请求耗时：10.85 / 7 = 1.55s。
- 平均单 action 耗时：10.85 / 12 = 0.9s。
- 客户端截图 + 开始上传 URL 平均耗时约 0.914s。
- NLU 分发 + 平台下发平均耗时约 0.658s。
- 云端调用 Agent 并完成结果下发，不含模型计算，平均约 0.3323s。

### 9.2 场景二耗时拆解

|分类|耗时|占比|
|---|---:|---:|
|总耗时|42s|100%|
|TTS（不含异步）|2s|4.8%|
|用户 query|8s|19%|
|客户端截图|1.847s|4.39%|
|上传 OSS 并获取 URL|4.611s|10.99%|
|Agent 模型|10.857s|25.83%|
|NLU 输出结果并下发|1.337s|3.1%|
|业务侧执行操作|5.596s|13.3%|
|平台下发 action 指令|2.392s|5.7%|
|等待|5.05s|12%|

### 9.3 性能瓶颈

主要耗时来源包括：

- Agent 模型推理，占比约 23%-26%。
- 用户 query 和二次确认，场景二中占比约 19%。
- OSS 上传和获取 URL，占比约 11%。
- 等待时间，占比约 12%。
- 业务侧动作执行，占比约 13.3%。

### 9.4 优化方向

#### 流程简化

- 去除最后一步发送后的反思，预计整体链路收益约 3s。
- 风险：如果发送按钮未实际点击成功，也可能播报“已发送”，需要产品确认是否接受。

#### VIVO 能力开放

- 如果 VIVO 开放“全选”“粘贴”等接口，搜索联系人和粘贴内容可各节省一步流程。
- 预计整体收益约 30%。

#### 等待时间优化

- 引入定制化端侧视觉小模型判断页面加载状态。
- 通过代码逻辑动态缩短等待。
- 让大模型额外输出等待时间，但依赖对应训练数据。

#### 截图上传优化

- 当前 IO 速度受网络影响，测试数据来自地库车辆实测，上线环境可能有所改善。
- 可优化截图压缩、上传链路、OSS URL 获取和平台转发。

#### 模型提速

- 替换云端模型为更小的 23BA2.6B MoE 模型，预计推理速度提高约 20%。
- 探索端侧 GUI Agent 模型，通过 token 压缩、图片压缩弥补芯片推理速度限制。
- 需要重新训练，周期较长。

## 10. 对话状态流转与任务状态继承

手车互联对话状态流转的目标是提升状态 NLU 继承能力。项目中需要维护 Agent 执行状态、Trigger 中断、Agent 未启动、异常状态以及记忆内容。

### 10.1 状态管理目标

- 判断 Agent 是否正在执行。
- 支持用户二次确认，例如“确认发送”。
- 支持多轮继承，例如上一轮 Agent 正在等待用户选择联系人。
- 支持 Trigger 打断，例如用户发起新指令。
- 支持异常状态恢复，例如页面异常、执行失败、连接断开。
- 支持 run store / history 记录，用于追踪每一步动作和状态。

### 10.2 典型状态

- Agent 未启动：普通语音请求进入 NLU 分发。
- Agent 执行中：当前任务处于镜像控制流程，后续用户输入可能需要继承到当前任务。
- 等待用户确认：如“确认发送吗？”。
- 等待用户澄清：如“找到多个联系人，你要发给第几个？”。
- Trigger 中断：新任务或高优先级指令打断当前 Agent。
- 执行失败：页面异常、action 无效、目标未找到、连接状态异常。
- 任务完成：finish 并清理或归档上下文。

### 10.3 记忆内容

建议记录：

- 用户原始 query。
- 当前任务目标。
- 当前生效 plan。
- 历史 action。
- 每轮截图 URL 或截图 ID。
- 每轮页面状态识别结果。
- 每轮模型输出 Thought / Action。
- 用户追问和确认结果。
- 异常原因。
- 是否已完成、是否中断。

这些信息既服务于多轮继承，也服务于问题追责、失败 case 回流和评测集构建。

## 11. 外部依赖与联调事项

### 11.1 VIVO 与小 V 对接

关键结论：

- 小 P 与小 V 对接方案选择“方案二”：小 P 作为意图分发和通道透传，具体槽位收集和补全由小 V 处理。
- 不推荐小 P 收集所有槽位后统一发给小 V，因为重名和查询实现复杂。
- 镜像场景下 2/3 屏和全屏是等比例适配，展示体验不理想，后续需要考虑与融合桌面应用整合。
- 当前不支持同时投射两个视频流，可能放到 V2/V3。
- 蓝牙连接断开后，需要清除通知消息，不能在车端缓存。

### 11.2 截图能力

截图链路是关键依赖：

- 5.x 可使用 `ScreenTaskManagerFactory` 接口。
- 6.0 截图能力处于开发中，预计特定排期后可联调。
- 已支持全屏、1/3、2/3 截图。
- 分屏截图有两个方案：
  - 定制截屏。
  - 整屏截图后根据业务状态进行切分。
- OCR 链路要求只包含手机镜像截图；若提供全屏截图，需要附带手机镜像位置坐标，由算法侧切分。

### 11.3 OCR 地址识别

OCR 地址识别方案：

- 云端部署 8B 多模态大模型。
- 未优化情况下单次推理约 0.7s。
- 输入来自系统截图服务上传的手机镜像截图。
- 输出图中地址信息给语音平台。
- 若导航无法直接解析完整地址，模型需要拆分出 POI name。
- 图片传递链路复用小 P 看世界：本地图片上传到阿里云 AI 小 P 专用 OSS，生成时效 URL 后供模型解析。

### 11.4 通知与 AI Bar

- 通知链路确认使用 System UI / AI Bar。
- 如果使用原生通知链路，会路由到 AI Bar 容器。
- 屏蔽/恢复指令发给 AI Bar，主要是显示操作，不影响业务侧真实通知。
- 等待与客户端联调。

### 11.5 导航能力

- 导航接口已确认支持完整地址搜索。
- 返回结果后默认取第一个。
- 若要支持用户选择候选结果，需要高德额外开发。
- V2 阶段需要拉导航同学评审。

## 12. 项目风险与待确认项

### 12.1 产品与体验风险

- V1 是否只做单轮，复杂多轮是否延期。
- 多轮是否由小 V 全权处理，还是小 P/平台/Agent 共同维护状态。
- 多轮是否引入二次确认。
- 发送消息后是否必须做最终反思确认，还是允许省略以换取速度。
- 打开/关闭手机应用如何与车机本地应用指令区分。
- 用户要发消息时，是否一定走搜索，还是在首页直接点击已有对话。

### 12.2 技术风险

- 微信和飞书 UI 定制化、弹窗、键盘差异导致泛化困难。
- 页面状态检测错误会导致 plan 走偏。
- 搜索结果 list 识别仍需进一步优化。
- 截图上传链路耗时和网络稳定性影响端到端体验。
- 真实 APP 动态页面可能导致点击位置失效。
- 多联系人、多群聊、无联系人等分支需要清晰的 recall / task_failed 规则。
- 粘贴输入依赖系统能力，若 VIVO 不开放更直接接口，会增加步骤和耗时。

### 12.3 联调风险

- XOS6 截图能力、jar 暴露、反射调用、5.x/6.0 差异。
- VIVO 镜像连接状态和音视频电话状态感知。
- 语音平台 action 下发与执行引擎端侧实现。
- OSS 上传、URL 生成、模型服务部署位置（PAI 或 Logan）。
- 通知 AI Bar 和业务侧链路对齐。

## 14. 项目价值与经验沉淀

### 14.1 技术价值

- 将 LLM/VLM 能力接入真实车机-手机镜像链路，实现从语音 query 到手机 APP 操作的闭环。
- 通过页面状态检测 + 通用 GUI Agent + 场景化增强，提升复杂 APP 自动化任务稳定性。
- 建立了 GUI Agent 动作空间、状态检测、追问澄清、反思重规划、执行后检测、数据回流的基础框架。
- 沉淀了微信、飞书、通知、导航等手车互联场景的算法和工程联调经验。

### 14.2 工程价值

- 打通 ASR/NLU、语音平台、车机端、截图服务、OSS、云端模型、CarlinkMirror 执行链路。
- 明确了各模块接口边界和联调依赖。
- 建立了耗时拆解方法，能够定位模型、截图、OSS、平台、业务执行和等待等耗时瓶颈。
- 为后续端侧 GUI Agent、轻量模型、动态等待、截图优化提供了依据。

### 14.3 数据与评测价值

- 页面状态检测从 73.48% 提升到 98.81%，多个子任务达到 100%。
- 输入框、键盘、弹窗、未读语音、对话框内容等局部检测任务形成了可复用评测方法。
- 数据构造覆盖人工采集、自动化生成、网络爬取和线上回流。
- 调研并沉淀了大量 GUI Agent 开源数据集，可用于通用能力训练和后训练。

## 15. 后续建议

1. 优先稳定微信发送消息闭环，明确 recall、task_failed、确认发送和最终 finish 的统一状态机。
2. 将页面状态检测、输入框有字、弹窗遮挡、联系人 list 识别纳入固定回归评测。
3. 梳理 Agent 上下文结构，显式区分历史轨迹、当前生效 plan、废弃 plan、当前 observation 和待执行 action。
4. 推动 VIVO 开放更直接的输入/粘贴/全选能力，减少长按和粘贴步骤。
5. 优化截图上传链路和等待策略，减少 OSS 与固定 sleep 对体验的影响。
6. 建立 hard case 回流机制，把失败 case 自动归因到页面识别、动作生成、执行链路、等待、用户澄清等类别。
7. 飞书方向优先补齐不同组织、不同首页、不同键盘、不同主题和弹窗数据。
8. 探索小模型页面状态检测或端侧视觉判断，用于动态等待和局部兜底。
9. 逐步引入更系统的 SFT / DPO / GRPO 数据，用于 action trajectory、反思纠错和任务成功率优化。
10. 在产品层面明确 V1/V2/V3 边界：V1 单轮发消息，V2 多轮澄清与通知控制，V3 更完整的 Multi-Agent / 端侧 Agent 能力。
