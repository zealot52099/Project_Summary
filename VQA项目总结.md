# VQA 项目总结

## 1. 项目背景与目标

VQA 项目围绕小鹏车载多模态问答能力建设，主要包含两条业务线：

- **小P看我**：基于车内 OMS 摄像头图像和座舱感知信息，回答用户关于车内人员、人数、位置、姿态、衣着、配饰、FaceID、儿童/宠物/物品等问题，并支持自然、口语化、多轮对话。
- **小P看世界**：基于车外图像或行车场景截图，回答车辆识别、车型/品牌/颜色、环境理解、地标/风景等问题，重点建设车外视觉理解与话术生成能力。

项目目标不是单点 VQA demo，而是建设一套可迭代的车载多模态问答系统，包括：

- 车端/云端链路打通。
- 端侧 VLM 或云端 VLM 能力验证。
- VQA 数据构造、清洗、增强和标注。
- SFT / LoRA / 融合训练。
- 多轮对话、FaceID、音区、座舱感知信息融合。
- 评测集、自动评测、产品打分和 badcase 回归。
- 安全审核、黄反/涉政/敏感问题拒答。
- 上线后数据回流和持续迭代。

## 2. 业务范围

### 2.1 小P看我

小P看我面向车内摄像头，主要回答用户关于车内状态和人员信息的问题。典型 query 包括：

- “现在车里有人吗”
- “现在车里有几个人”
- “看看车里有几个人”
- “小P看我在干什么”
- “车里的人都在干什么”
- “车里有几个人，我在干什么”
- “副驾有人吗”
- “后面的人是谁”
- “老王的衣服怎么样”
- “老王坐在哪儿”
- “我这身衣服适合明天去广州吗”

能力范围包括：

- 车内人数识别。
- 座位位置识别：主驾、副驾、二排左、二排中、二排右等。
- 性别、年龄、儿童/成人判断。
- 发型、衣着、配饰、眼镜、帽子、挂绳等描述。
- 姿态识别：玩手机、开车、聊天、吃东西、睡觉、打电话等。
- FaceID 昵称融合。
- 多轮指代继承，例如“那他呢”“旁边的人呢”。
- 拒识和安全边界处理。
- 摄像头遮挡、关闭、无人位置等负例场景处理。

### 2.2 小P看世界

小P看世界面向车外图像理解，重点在车辆识别和场景描述。典型能力包括：

- 识别前方车辆品牌、车型、颜色。
- 识别小鹏车型和其他品牌车型。
- 识别车辆相对方位，例如正前方、左前方、右前方。
- 根据用户自然问法生成小P风格回答。
- 支持车外风景、地标、道路场景等通用视觉问答扩展。

数据和训练设计分为两个阶段：

1. **基础视觉能力阶段**：输入图片和结构化问题，例如“请识别这辆车的品牌、车型和颜色”，输出结构化结果。
2. **话术风格生成阶段**：输入图片和口语化 query，例如“小P看看前面是什么车”，输出符合小P语气的自然回复。

## 3. 工程链路

### 3.1 小P看我链路

XOS6.0 小P看我链路围绕 NLU 分发、VLMU、车内图像、感知信息、TTS 和 AI 小P客户端展开。

整体流程：

1. 用户在车内音区发起 query。
2. 云端 NLU 识别是否命中小P看我意图。
3. 对车内/车外音区、车型、功能开关进行前置校验。
4. VLMU 获取车内 OMS 图像和感知信息。
5. 系统 prompt 拼接用户 query、音区、车内感知信息、图像占位内容等。
6. 调用 VLM 生成回答。
7. 后处理模型输出，进行安全审核、话术约束或格式修正。
8. 调用 AI 小P客户端和 TTS 播报。

前期方案采用云端 NLU 分发，部分难例依赖模板/白名单；终态期望端上 VLM 支持分发。V3 主要考虑云端分发和基础 query，如“车里有几个人”；V4 再考虑更复杂场景、多轮能力和更多 CV 小模型融合。

### 3.2 小P看我校验与兜底逻辑

链路中定义了多级校验：

- **音区校验**：车内音区命中意图则继续；车外音区默认拦截，提示“车外未开放这个功能，请在车里问我吧”。
- **车型校验**：车型支持则继续，不支持则提示“咱们车还没有这个功能哦”。
- **功能开关校验**：功能开启则继续，关闭则提示“请先打开小P看我应用开关哦”。
- **OMS 图像获取**：3 秒内成功则继续，失败时可尝试使用座压信号，系统异常则提示稍后再试。
- **座压信号获取**：3 秒内成功则作为补充判断，失败则拦截。
- **图像 + 座压融合**：
  - 图像无人 + 座椅无人：无人场景。
  - 图像无人 + 座椅有人：有人场景。
  - 图像有人 + 座椅无人：有人场景。
  - 图像有人 + 座椅有人：有人场景。

### 3.3 小P看世界链路

小P看世界当前基座模型为 **Qwen3-VL-8B**，部署时需要申请 OSS 权限，将模型上传到 OSS 测试路径，再进行部署。

现有链路基于 XGPT / VQA 能力改造，后续计划补齐 600 自研模型链路。推理服务通过 OpenAI API 风格接口调用，支持本地 vLLM 服务和 PAI 平台测试环境服务。

### 3.4 TTS 与流式体验

600 v5 采用流式链路，目标是：

- 3 秒以内生成垫话回复。
- 6 秒以内生成完整话术回复。

候选方案包括：

1. 复用语音主链路，VQA AP/AF + actionName + cmd + TTS 垫话。
2. VLMU 流式交互，先由 xpthink 生成垫话，再由 VLM 生成正式回答。

安全审核链路采用方案 2。

### 4.2 小P看世界模型

小P看世界使用 Qwen3-VL-8B 作为当前基座。训练目标包括：

- 品牌识别。
- 车型识别。
- 颜色识别。
- 位置/方位识别。
- 车辆描述和小P风格回答。

训练策略包括：

#### 方案 A：顺序训练

先用所有数据训练阶段一，直到视觉基础能力收敛；再冻结视觉编码器或降低学习率，用所有数据训练阶段二话术风格能力。

优点是简单稳定，能保证模型先“看准”。

#### 方案 B：交替训练

在每个 epoch 或每 N 个 batch 后交替使用结构化任务和风格化任务。例如一个 batch 学结构化车辆识别，下一个 batch 学口语化回答。

优点是同时学习“看准”和“说好”，可能产生更好协同。

#### 方案 C：混合训练

直接将两种数据混合成一个训练集，让模型自己学习任务关联。优点是实现简单。

### 5.2 分类模型与聚类

分类模型对比：

- `clip-vit-large-patch14-336 + 线性分类头`：验证准确率约 0.55。
- `chineseclip-vit-large-patch14-336px + MLP`：验证准确率约 0.83。
- `clip-vit-large-patch14-336 + MLP`：top1 约 0.72。
- `clip-base-patch32 + MLP`：top1 约 0.73。
- `clip-base-patch16 + MLP`：top1 约 0.73。

考虑推理速度，最终选择 **clip-base-patch32** 作为特征提取模型。

在 **4300w 新增图片** 上进行分类，样本总数为 **43,836,197**。

聚类方案：

- 方法：K-means + Faiss。
- 先按分类体系打为 21 类。
- 对每个类别进行第一轮聚类，k=5。
- 对第一轮聚类后数量仍超过 50w 的桶再次聚类，k=5。
- 两轮聚类后得到 **258 个桶**，每个桶图片数量控制在 50w 以下。

最终 pipeline 目录为：

`/workspace/zhanghj15@xiaopeng.com/fuyao/data_process/img_classify`

主要工具包括：

- `clipfeature_infer.py`：推理图像 CLIP 特征。
- `clip_feature_classify.py`：用分类头对 CLIP 特征打标。
- `clip_distribution.py`：查看特征类别分布。
- `clip_cluster.py`：按类别聚类。
- `train_classhead.py`：训练 CLIP 分类头。
- `mv_h5.sh`：筛选小于 50w 的簇。
- `read_h5_clip.py`：读取 CLIP 特征。

### 6.2 无人位置负例

基于图片中人物位置，挑选无人位置进行询问。例如图片只有主驾有人，用户问“二排座位有人吗？”，模型应回答无人。

典型回复：

- “小P没看到人哦！”
- “小P瞅了半天没看到人呀！是在和小P玩捉迷藏吗？”

### 6.3 摄像头关闭与遮挡负例

摄像头遮挡数据通过修改图片为大面积遮挡来模拟。回复统一采用兜底话术，例如：

- “摄像头好像被遮挡了，小P什么也看不到呢。”
- “系统遇到异常，请稍后再试一下吧。”

关闭摄像头负例通过链路干预实现。

### 6.4 敏感话术数据

敏感问题统一设置 query_type 为“敏感问题”，兜底话术：

### 6.8 FaceID 数据

620 新增 FaceID 场景，输入额外包含 FaceID 信息，例如：

```json
{"主驾":"小王","副驾":"小张","二排右":"小李"}
```

典型 query：

- “后面的人是谁？”
- “老王的衣服怎么样？”
- “老王坐在哪儿？”

FaceID 逻辑包括：

- 位置无人：走无人兜底。
- 位置有人且全部未注册：引导注册。
- 位置有人且部分注册：使用昵称话术，不提及未注册的人。
- 人脸评价中如果有 FaceID，回复中位置可替换成昵称。

### 6.9 多轮继承数据

620 NLU 不再进行改写，VLM 需要实现“那他呢”等指代继承。

多轮数据包括：

- 首轮提及位置或昵称，后续问“他”。
- 同一人问答，无明确指代，但期望继承上文特定人员。
- 多轮 VQA 不继承数据，即历史轮与当前问题无关时不能误继承。
- 业务范围内多轮：人脸性格、人脸评价、发型搭配、明星人脸、服装搭配。
- 业务范围外多轮：历史轮穿插宠物、敏感、玩具、奥特曼等单轮对话，最后一轮落到业务范围内任务。

日报中记录：

- 构造 1738 条指代多轮数据 + 1738 条昵称多轮数据。
- 业务范围内多轮 6000 条。
- 业务范围外多轮 6000 条。
- 纠错对话 3000 条。

### 6.10 复杂对话：着装适合天气

天气穿衣类数据构造流程：

1. 生成天气 query，例如“明天广州天气如何”。
2. 跑批获取天气 API 结果。
3. 基于天气 query、城市和时间生成 VQA query，例如“我这身衣服适合明天去广州吗”。
4. 结合天气回复和衣着信息生成回复话术。
5. 将天气问答作为历史轮，VQA 穿衣建议作为下一轮，构造多轮对话数据。

数据优化点包括：

- 统一增加天气和气温。
- 下装看不见时不要强行描述。
- 温差大时建议带外套穿脱。
- 雨雪天气建议带雨具。
- 支持 FaceID 昵称。
- 炎热天气数据生成。

### 6.12 动作姿态数据

动作姿态数据基于 MotionBenchCaption-train：

Pipeline：

1. 数据下载。
2. 视频抽帧。
3. 图片 resize。
4. 数据翻译。
5. 数据风格改写。
6. 根据描述生成问答对。
7. 写入 Excel 便于人工 review。
8. 人工 review 重新标注。

### 7.2 话术风格改写

目标是让模型回答更自然、口语化、像朋友闲聊，而不是机械问答。

要求包括：

- 自然对话流。
- 话题多样性。
- 上下文连贯。
- 信息准确。
- 一问一答配对。
- 生成 3-5 轮对话。
- 每句话控制在 50 字以内。
- 助手回答先给结论，必要时补充解释。
- 可适当延伸，但避免过度情绪化和制造强情感连接。

评估发现：

- qwen3omni、doubao、gpt5.2 生成的第一句大量是“图片里有什么”，多样性不足。
- doubao 部分回答前后逻辑不连贯。
- qwen3omni 偶发幻觉，回答较简洁，有时偏机械。
- GPT5.2 通常更稳，但也存在视觉描述依赖 caption 本身错误的问题。

## 11. 安全与拒识

安全相关包括：

- 敏感问题拒答。
- 涉政数据。
- 黄反数据。
- FaceID 攻击数据。
- 儿童隐私和面部模糊场景。
- 外貌评价边界。
- 年龄、身材等敏感词规避。

系统 prompt 要求：

- 回复礼貌、口语化。
- 符合社会主义价值观。
- 外貌、衣着问题尽量正向夸奖。
- 可夸外搭，不描述内搭。
- 车内有老人时，不称呼“老人”，改称成年男士/女士。
- 描述性格以温柔、开朗、亲切为主。

安全数据后续待验证：

- 是否将安全数据混入训练。
- 若影响主能力较大，则需谨慎。
- 可加入一键禁用 VQA 的配置选项，从 NLU 侧规避可能触发舆情的问题。

## 13. 主要问题与风险

### 13.1 感知与图像质量风险

- OMS 图像夜间为黑白，颜色描述容易错误。
- 摄像头遮挡、关闭、低清晰度会导致幻觉。
- 人体 bbox 和 seat_label 错误会传导到 VQA 回复。
- 混色衣服、配饰、发型等细节依赖标注质量。

### 13.2 多轮与指代风险

- “他”“他们”“旁边的人”“左边的人”等指代容易错。
- 多轮需要判断是否继承历史，如果不该继承却继承，会回答错对象。
- FaceID 与位置、昵称、未注册用户之间的逻辑较复杂。

### 13.3 话术风险

- 回答机械，第一句总是“图片里有什么”。
- 语气词过多或过度“小P化”。
- 遮挡、儿童闭眼等场景可能出现不合适描述。
- 外貌评价、年龄、身材、隐私类话术需要约束。

### 13.4 数据风险

- 自动生成数据可能实体不在画面中。
- caption 错误会污染训练数据。
- 低质量抽帧会影响动作姿态训练。
- 开源数据存在英文、领域偏差、商用限制、下载失败等问题。
- 分类与聚类策略需要兼顾类别均衡和类内多样性。

### 13.5 评测风险

- 产品打分存在主观性。
- 每轮评测标注 case 和标注人变化会导致波动。
- 自动评测依赖 GT 和语义相似度，生成式答案难以完全客观衡量。
- 量化模型评测中，部分大模型 API 在 do_sample=False 时仍存在输出不稳定问题。

## 14. 后续建议

1. **统一数据版本管理**：每版训练数据都记录来源、规模、query_type、生成脚本、过滤规则和问题修复点。
2. **强化 badcase 闭环**：将产品低分、线上错误、自动评测失败统一归因到感知、数据、prompt、模型、后处理和安全类别。
3. **补齐多轮评测**：重点覆盖“他/他们/旁边/左边/右边/那个人”等指代场景，以及不应继承历史的多轮负例。
4. **提升物品识别能力**：针对手机、帽子、包包、水杯、眼镜、口罩、耳机等高频物品补充数据。
5. **优化儿童和隐私场景话术**：避免“神态安详”等不合适表达，儿童面部、隐私和敏感评价要有专门规则。
6. **完善 FaceID 融合逻辑**：明确无人、未注册、部分注册、多人的话术模板和模型输入格式。
7. **小P看世界车型识别专项优化**：重点提升其他品牌车型、小鹏 P5、新 P7、G3 等低准确率车型。
8. **加强对抗数据**：包括无人位置、遮挡、摄像头关闭、错误属性、图中不存在对象、涉政黄反、FaceID 攻击等。
9. **引入分类 + 聚类采样**：在保证类别覆盖的同时提升类内多样性，缓解头部类别和典型样本偏置。
10. **稳定评测体系**：固定核心评测集，区分产品主观评分、自动指标、badcase 回归和公开 benchmark。
11. **探索课程学习和 RL/DPO**：先训练视觉基础，再训练话术风格，再针对 badcase 和偏好数据做后训练。
12. **优化端侧体验**：关注 3s 垫话、6s 完整回复目标，结合流式链路、缓存和后处理减少用户等待。
